# API Reference

The Kapp API surface is generated from two sources:

1. **REST / OpenAPI** — registered chi routes in
   `services/api/routes.go`; the running service serves a generated
   spec at `/api/v1/openapi.json` (handler:
   `services/api/openapi.go::openAPIHandler.serve`).
2. **gRPC** — protobuf definitions under `proto/kapp/v1/`; generated
   code in `gen/` (regenerated by `make proto-gen`).

This document is meta — it explains *how* the reference is generated
and *what conventions* every endpoint must follow. The browsable
reference is at `/api/v1/docs` (Redoc) and `/api/v1/openapi.json`
(raw spec).

Cross-references:

- API versioning policy: [API_VERSIONING.md](./API_VERSIONING.md)

## 12.1 OpenAPI Generation

The OpenAPI 3.0 spec is generated **at runtime** from the registered
KTypes in `internal/ktype/openapi.go::GenerateOpenAPISpec`. Every
KType registered in the `ktypes` table contributes:

- Schema definition (from the KType `schema` JSONB)
- CRUD endpoints under `/api/v1/records?ktype=<name>` and
  `/api/v1/records/<name>/{id}`
- Field constraints (required, format, length) mirrored from the
  KType definition
- Example payloads from the `examples` block of the KType

Static endpoints (auth, users, plans, etc.) are described in the
manually maintained companion `internal/ktype/static_openapi.go`
(planned) — until then they appear in the spec without per-field
schemas. A CI gate (`.github/workflows/api-versioning-check.yml`)
ensures the OpenAPI shape is compatible across releases.

Serve / inspect:

```bash
# JSON spec
curl -s -H "Authorization: Bearer $ADMIN_TOKEN" \
  https://api.kapp.example.com/api/v1/openapi.json | jq

# Browsable UI (Redoc, served from /api/v1/docs)
open https://api.kapp.example.com/api/v1/docs
```

## 12.2 Endpoint Documentation Conventions

Every endpoint documented in the spec includes:

| Field                  | Source                                                            |
| ---------------------- | ----------------------------------------------------------------- |
| HTTP method + path     | chi route registration in `services/api/routes.go`                |
| Required headers       | `Authorization: Bearer <jwt>` always; `X-Idempotency-Key` on mutations (POST/PUT/PATCH) — see `internal/idempotency`. |
| Request body schema    | JSON Schema, derived from the KType definition for `/records/...` routes; hand-written for static routes. |
| Response schema        | Success and structured error responses (`{ "error": { "code": "...", "message": "..." } }`). |
| Rate-limit behaviour   | Per-tenant + per-IP limits. Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`. |
| Feature flag           | Set by `tenant_features`. Endpoints requiring a flag return 402 Payment Required when the tenant's plan doesn't include the feature. |
| Example curl           | One per non-trivial endpoint, included via the OpenAPI `x-codeSamples` extension. |

### Example endpoint documentation

```http
POST /api/v1/records
Authorization: Bearer <jwt>
X-Idempotency-Key: <unique id>
Content-Type: application/json

{
  "ktype":         "crm.deal",
  "ktype_version": 3,
  "data": {
    "name":  "Acme Q2 expansion",
    "value": 50000,
    "stage": "qualification"
  }
}
```

```http
HTTP/1.1 201 Created
Location: /api/v1/records/<uuid>
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 999
X-RateLimit-Reset: 1735689600

{
  "id":            "<uuid>",
  "tenant_id":     "<uuid>",
  "ktype":         "crm.deal",
  "ktype_version": 3,
  "data":          { "...": "..." },
  "version":       1,
  "created_at":    "2026-05-25T06:49:00Z",
  "created_by":    "<uuid>",
  "updated_at":    "2026-05-25T06:49:00Z"
}
```

Errors follow a stable structure (planned spec at
`internal/platform/httperror.go`):

```json
{
  "error": {
    "code":      "validation.required",
    "message":   "field 'name' is required",
    "path":      "/data/name",
    "request_id":"a1b2c3..."
  }
}
```

## 12.3 Webhook Payload Documentation

Webhook payloads are versioned alongside the REST API. Per-event
payload schemas live next to the event constants:

- Outbound webhook delivery contract: `internal/webhooks/delivery.go`
  emits payloads matching the schemas in
  `docs/webhooks/<event>.json` (planned).

Headers on every webhook delivery:

| Header               | Value                                                         |
| -------------------- | ------------------------------------------------------------- |
| `X-Kapp-Event`        | Event name (e.g. `record.created`)                            |
| `X-Kapp-Event-Id`     | UUID, unique per delivery (used for idempotency)              |
| `X-Kapp-Signature`    | `sha256=<hex>` HMAC over the body with the webhook secret      |
| `X-Kapp-Timestamp`    | Unix epoch seconds; reject deliveries older than 5 minutes    |
| `User-Agent`          | `kapp-webhook/<version>`                                      |

Verification snippet (Go):

```go
mac := hmac.New(sha256.New, []byte(webhookSecret))
mac.Write([]byte(timestamp + "." + string(body)))
expected := "sha256=" + hex.EncodeToString(mac.Sum(nil))
if !hmac.Equal([]byte(expected), []byte(r.Header.Get("X-Kapp-Signature"))) {
    return errors.New("invalid signature")
}
```

## 12.4 gRPC Service Documentation

`proto/kapp/v1/*.proto` is the single source of truth. Generated
documentation (HTML / markdown) is produced by `buf generate`:

```bash
# Regenerate Go bindings + reflection (idempotent):
make proto-gen

# Run linter:
make proto-lint

# Check for breaking changes vs. main:
make proto-breaking
```

CI runs all three on every PR (`.github/workflows/buf.yml`).

Comments in proto files become the public documentation. Convention:

```protobuf
// Records is the canonical CRUD service for KRecord-backed objects.
// All mutations are idempotent if X-Idempotency-Key (REST) /
// idempotency_key (gRPC) is provided.
service Records {
  // Create a new record. Returns NotFound if the KType is unknown,
  // FailedPrecondition if the tenant's plan does not include the KType.
  rpc Create(CreateRequest) returns (CreateResponse);
}
```

gRPC-Gateway translates the same service into REST via the
annotations in the proto files (see `services/api/grpc.go`).

## 12.5 Authentication

All authenticated endpoints accept a JWT in `Authorization: Bearer
<token>`. Tokens are signed HS256 by default (`internal/auth/jwt.go`),
with RS256 supported via `KAPP_JWT_ALGORITHM=RS256` and a key ring
fetched from the configured secret provider.

Claims:

| Claim       | Purpose                                                   |
| ----------- | --------------------------------------------------------- |
| `iss`       | `kapp` (configurable via `KAPP_JWT_ISSUER`)               |
| `aud`       | `kapp` (configurable via `KAPP_JWT_AUDIENCE`)             |
| `sub`       | User ID                                                   |
| `tenant_id` | Tenant the token is scoped to                              |
| `exp`       | Access token: 15 min (override `KAPP_JWT_ACCESS_TTL`)     |
| `iat`       | Issued-at                                                 |
| `roles`     | Comma-separated role names; resolved against `roles` table |

Refresh flow:

```http
POST /api/v1/auth/refresh
Authorization: Bearer <refresh-token>

→ HTTP/1.1 200 OK
{
  "access_token":  "<new access token>",
  "refresh_token": "<new refresh token>",
  "expires_in":    900
}
```

Refresh tokens are single-use (the refresh response invalidates the
old refresh token and issues a new one). The chain is stored in
`sessions` (`migrations/000013_auth_sessions.sql`).

## 12.6 Client SDKs

Two SDK directions:

1. **TypeScript SDK** — `packages/client`. Hand-written facade around
   `fetch` with typed shapes from the OpenAPI generator.
   Consumed by `apps/web`. Versioned to match the API.
2. **Rust SDK** — `packages/sdk-rs`. Builds the generated client from
   the gRPC proto files. Used by external plugins and the CLI tools.

Both SDKs ship in the public release packages; see CI workflow
`supply-chain.yml` for the SLSA attestations.

## 12.7 Versioning Summary

See [API_VERSIONING.md](./API_VERSIONING.md) for the full policy. In
short:

- **Path-versioned** (`/api/v1`). Breaking changes go to `/api/v2`.
- **OpenAPI-checked**: `api-versioning-check.yml` workflow compares
  the new spec against `main` and fails on incompatible changes
  (removed fields, changed required-ness, narrowed types).
- **Sunset header**: deprecated endpoints emit
  `Deprecation: true` and `Sunset: <RFC1123 date>` headers ≥ 90 days
  before removal.
